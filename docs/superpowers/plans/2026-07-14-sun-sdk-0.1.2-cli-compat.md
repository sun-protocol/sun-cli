# Sun SDK 0.1.2 CLI Compatibility Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish `@sun-sdk/*@0.1.2` with the compatibility capabilities required for `sun-cli` to remove `@sun-protocol/sun-kit`.

**Architecture:** Keep protocol math and transaction construction inside `sun-sdk`; expose CLI-compatible convenience inputs only at SDK boundaries. Runtime owns generic ABI calls and balance reads, API owns SunPump HTTP network support, and protocols own V2/V3/V4 compatibility derivation.

**Tech Stack:** TypeScript, pnpm workspace, Vitest, tsup, `@sun-sdk/core`, `@sun-sdk/chains`, `@sun-sdk/runtime`, `@sun-sdk/api`, `@sun-sdk/protocols`, `@sun-sdk/sunswap-v2`, `@sun-sdk/sunswap-v3`, `@sun-sdk/sunswap-v4`.

## Global Constraints

- Work in `/Users/caleb/Documents/code/tron/sdk/sun-sdk-design/sun-sdk`.
- Package target is `@sun-sdk/*@0.1.2`.
- Root and published package engines must support Node `>=20`.
- SunPump must support Nile API base URL `https://tn-api.sunpump.meme/pump-api`.
- CLI compatibility APIs must preserve existing `sun-cli` behavior; new richer SDK fields must not force CLI output changes.
- Existing SDK APIs remain source-compatible unless a test proves the current signature is unusable.
- Do not add a sun-kit dependency to SDK.
- Do not use real private keys in SDK unit tests.
